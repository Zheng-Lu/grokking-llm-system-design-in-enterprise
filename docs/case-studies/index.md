# Case Studies

Case studies are the evidence layer of this book. Each chapter takes public material and turns it into a reusable system design narrative.

## How this section is organized

- every chapter follows the same frame: problem, why naive solutions fail, architecture, trade-offs, failure modes, and sources
- chapters should cross-link to relevant handbook sections when that helps readers generalize the lesson
- some chapters begin as starter scaffolds and become fully source-backed over time; status should stay explicit

## What a strong case study contains

A strong case study should:

- teach a reusable enterprise design lesson
- make the core constraint visible early
- separate sourced facts from inference
- explain the trade-offs, not only the architecture

## How to contribute one

Use the chapter template in [Contributing / Chapter Template](../contributing/chapter-template.md), then make sure the resulting page is reflected in both the navigation and this section overview.

## Editorial rule

Case studies do not need perfect visibility into every implementation detail. They do need honest boundaries between fact, inference, and open questions.
