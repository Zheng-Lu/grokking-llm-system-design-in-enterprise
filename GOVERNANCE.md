# Governance

## Goals

The project should stay:

- technically rigorous
- easy to contribute to
- lightweight to maintain
- opinionated about quality without becoming bureaucratic

## Roles

### Contributors

Contributors propose issues, improve documentation, submit pull requests, and help refine chapter quality.

### Reviewers

Reviewers focus on technical clarity, structure, citation quality, and whether a change improves the book's usefulness.

### Maintainers

Maintainers steward direction, manage releases, make final merge decisions, and keep the editorial bar consistent across the repository.

## Decision making

- Prefer discussion and technical consensus.
- If consensus does not emerge, maintainers decide.
- Major scope or structural changes should be documented in an issue or pull request before implementation.

## Merge policy

Maintainers may merge a pull request when it is:

- in scope for the project
- structurally sound
- readable and technically useful
- sufficiently sourced where it makes factual claims

## Editorial bar

The project values:

- concise writing
- explicit trade-offs
- enterprise realism
- reusable patterns over vendor hype

Changes that add noise, weak sourcing, unnecessary abstraction, or unmaintainable customization should be rejected or narrowed.

## Maintainer expectations

Maintainers should:

- review with clarity and respect
- avoid silent rewrites of contributor intent
- explain non-obvious rejections
- keep process overhead low

## Project evolution

This governance model is intentionally lightweight for MVP. If the contributor base grows, the project can add clearer maintainer rotation, release ownership, or chapter-area stewardship later.
